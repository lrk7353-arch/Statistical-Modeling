# 统计建模项目启动文档 v1.0

生成日期：2026-04-26  
项目方向：公共数据与数据要素  
建议题目：**公共数据“沉睡资产”的识别、成因诊断与场景化激活推荐研究——以上海公共数据开放平台为例**

---

## 一、项目定位

### 1.1 研究背景

公共数据开放已经从“有没有数据”进入“数据能不能被真正用起来”的阶段。许多城市开放平台积累了大量数据集，但现实中可能存在三类问题：

- 数据已经开放，但下载、浏览或接口调用很低。
- 数据质量不差，甚至具有治理价值，却没有形成明显使用。
- 单个数据集价值有限，但如果与其他数据组合，可以服务交通、养老、城市安全、营商环境、公共卫生、生态环境等治理场景。

因此，本项目不做普通的“公共数据开放质量评价”，而是聚焦一个更具体的问题：

> 哪些公共数据不是简单没人用，而是“本应有用却没有被充分使用”？为什么没有被用起来？如何通过质量改进、场景绑定和数据组合推荐实现激活？

### 1.2 核心创新

本项目以“沉睡资产”为主问题，但不把沉睡资产简单定义为下载量低。我们将其定义为：

> **具有较高潜在使用价值，但实际使用表现显著低于预期的公共数据资源。**

核心创新包括：

- 用“预测潜力 - 实际使用”的使用残差识别高潜低用数据。
- 用质量诊断和模型解释分析沉睡成因。
- 用异质图学习和语义匹配推荐场景化数据组合。
- 用反事实模拟评估 API 化、补充字段说明、提高更新频率、加入专题数据包等激活策略的预期收益。

### 1.3 项目最终目标

形成一套面向公共数据平台的数据资产运营框架：

1. 识别哪些数据集是高潜低用的“沉睡资产”。
2. 诊断沉睡原因：质量问题、接口问题、语义问题、组合问题、场景问题。
3. 推荐可执行的激活路径：质量修复、接口化、场景绑定、数据组合包。
4. 给出公共数据开放平台优化建议，服务数据要素价值释放和城市治理应用。

## 二、要解决的关键问题

### 2.1 研究问题

本项目围绕五个问题展开：

**RQ1：公共数据的使用表现受哪些因素影响？**  
分析数据主题、部门、格式、更新频率、API、字段说明、空间属性、时间属性、语义清晰度等因素对下载量、浏览量、接口可用性和应用关联的影响。

**RQ2：哪些数据集属于“高潜低用”的沉睡资产？**  
先建立数据集理论使用潜力模型，再将预测潜力与实际使用表现比较，识别潜力高但实际低的数据集。

**RQ3：沉睡资产为什么沉睡？**  
从元数据质量、内容质量、机器可读性、接口可用性、更新时效、语义可理解性、跨数据集可组合性等角度进行成因诊断。

**RQ4：沉睡数据如何被场景化激活？**  
构建治理场景库和数据集异质图，推荐可组合的数据包，例如养老服务、交通治理、城市安全、公共卫生、生态环境等场景所需数据组合。

**RQ5：不同激活策略的预期收益有多大？**  
通过反事实模拟和相似成功案例匹配，评估“增加 API”“补充字段说明”“提高更新频率”“转换机器可读格式”“加入专题数据包”等措施对使用表现的潜在提升。

### 2.2 不解决的问题

为避免题目虚化，本项目明确不声称：

- 不声称能直接证明某个数据集被政府政策决策采用。
- 不把下载量直接解释为治理采用。
- 不把所有低下载数据都定义为沉睡资产。
- 不做纯管理学式指标评价。

本项目研究的是：

> 公共数据资源的可观测使用表现、潜在使用价值、质量缺陷、组合价值和激活路径。

## 三、总体技术框架

项目采用“画像 - 诊断 - 识别 - 分型 - 推荐 - 模拟”的六层框架。

### 3.1 第一层：公共数据资源画像

从上海公共数据开放平台采集或整理数据集元信息，形成数据集主表。

建议字段：

- 数据集 ID
- 数据集名称
- 数据集摘要
- 所属部门
- 主题分类
- 标签关键词
- 文件格式
- 是否机器可读
- 是否提供 API
- 更新时间
- 更新频率
- 下载量
- 浏览量
- 数据表数量
- 字段数量
- 字段说明完整度
- 是否包含空间字段
- 是否包含时间字段
- 是否关联典型应用或数据产品
- 数据样例可访问性

形成基础表：

- `dataset_master`：数据集主表。
- `field_master`：字段信息表。
- `usage_metrics`：下载、浏览、接口等使用表现表。
- `quality_features`：质量诊断特征表。
- `scenario_library`：治理场景库。
- `graph_edges`：异质图边表。

### 3.2 第二层：开放质量自动诊断

质量诊断分为元数据质量和内容质量。

元数据质量诊断：

- 标题是否过短或含义模糊。
- 摘要是否缺失或过短。
- 字段说明是否完整。
- 更新时间是否过久。
- 更新频率是否异常。
- 文件格式是否可机读。
- 主题分类是否模糊。
- 是否存在同部门重复数据。
- 是否存在跨部门语义重复数据。

内容质量诊断：

- 缺失率。
- 重复记录比例。
- 字段类型异常。
- 编码不一致。
- 时间字段不规范。
- 空间字段不可解析。
- 数值异常。
- 字段命名混乱。

输出：

- 元数据质量指数。
- 内容质量指数。
- 可机读性指数。
- 时效性指数。
- 语义清晰度指数。
- 可组合性指数。

### 3.3 第三层：高潜低用沉睡资产识别

本项目不把低下载直接当作沉睡，而是使用“理论使用潜力 - 实际使用表现”的残差框架。

实际使用表现：

```text
ActualUse_i = 标准化(log(下载量 + 1), log(浏览量 + 1), API可用性, 应用关联情况)
```

理论使用潜力：

使用数据集的内在价值特征建模，包括：

- 治理场景相关性。
- 主题公共价值。
- 字段丰富度。
- 空间字段和时间字段能力。
- 可组合性。
- 语义清晰度。
- 部门数据供给能力。
- 同类数据集历史使用表现。

推荐建模方式：

- LightGBM / CatBoost 预测潜在使用水平。
- 只用“内在价值特征”训练潜力模型，尽量不把无 API、格式差、更新慢等使用障碍直接纳入潜力模型。
- 质量和接口特征主要用于后续成因诊断，避免模型把“因为质量差所以没人用”解释为“本来就不该有人用”。

沉睡度：

```text
DormantScore_i = PredictedPotentialUse_i - ActualUse_i
```

沉睡资产判定：

- 预测潜力位于前 30%。
- 实际使用位于后 50%。
- 或 DormantScore 显著为正。

输出：

- 高潜低用数据集清单。
- 沉睡度排序。
- 各主题、部门、场景下的沉睡资产分布。

### 3.4 第四层：沉睡资产成因分型

使用聚类、BGMM、层次聚类或 HDBSCAN 对沉睡资产进行分型。

建议类型：

1. **高质量低曝光型**  
   数据质量较好，但浏览和下载很低，可能缺少场景绑定和平台推荐。

2. **关键场景缺 API 型**  
   数据具有治理场景价值，但没有 API 或接口能力弱，使用门槛高。

3. **更新滞后型**  
   数据主题重要，但更新时间过久或更新频率不稳定。

4. **语义说明不清型**  
   标题、摘要、字段解释不足，用户难以理解数据用途。

5. **孤岛数据型**  
   单个数据集价值有限，缺少与其他数据集的连接和组合推荐。

6. **可组合但未组合型**  
   与多个治理场景和数据集存在语义或字段连接，但平台未形成专题数据包。

成因解释方法：

- SHAP 解释使用潜力和实际使用模型。
- 残差分解解释沉睡度来源。
- 同类高使用数据集对照。
- 人工抽样核验典型沉睡数据集。

### 3.5 第五层：场景化组合推荐

构建治理场景库，并用异质图学习或语义匹配推荐数据组合。

治理场景库建议包括：

- 养老服务。
- 交通治理。
- 城市安全。
- 营商环境。
- 公共卫生。
- 生态环境。
- 教育资源。
- 文旅消费。
- 社区服务。
- 应急管理。

异质图节点：

- 数据集。
- 字段。
- 部门。
- 主题。
- 治理场景。
- 典型应用。
- 数据产品。

异质图边：

- 数据集属于部门。
- 数据集属于主题。
- 数据集包含字段。
- 数据集与场景语义匹配。
- 数据集与数据集字段相似。
- 数据集与数据集空间字段可连接。
- 数据集与数据集时间字段可连接。
- 数据集被应用关联。

推荐方法：

- 文本 embedding：SBERT/BGE/text2vec。
- 数据集相似度：标题、摘要、字段、主题综合相似度。
- 规则约束：空间字段、时间字段、主体字段是否可连接。
- 图方法：Node2Vec、GraphSAGE、异质图 link prediction。
- 推荐目标：为沉睡数据集推荐可组合数据集和治理场景。

示例组合：

- 养老服务：老龄人口 + 养老机构 + 医疗机构 + 公交站点 + 社区服务设施。
- 交通治理：道路设施 + 事故点 + 施工信息 + 公交地铁 + 停车资源。
- 城市安全：应急避难场所 + 消防站 + 人口密度 + 道路网络 + 重点单位。
- 公共卫生：医疗机构 + 药店 + 社区人口 + 公共服务设施 + 交通可达性。
- 生态环境：空气质量 + 绿地 + 水体 + 建设用地 + 人口暴露。

### 3.6 第六层：策略模拟与激活路径

针对不同沉睡类型，设计可执行的激活策略。

策略库：

- 增加 API。
- 转换为 CSV/JSON 等机器可读格式。
- 补充字段说明和数据样例。
- 提高更新频率。
- 与治理场景绑定。
- 纳入专题数据包。
- 推荐与高相关数据集组合。
- 将相似重复数据合并或建立索引。

模拟方法：

1. **反事实模拟**  
   在预测模型中改变可行动变量，例如将 `api_flag` 从 0 改为 1、将更新频率提高一级，观察预测使用表现变化。

2. **相似成功案例匹配**  
   为沉睡数据集寻找相似但高使用的数据集，对比差异，提取可迁移的激活策略。

3. **分型策略匹配**  
   不同沉睡类型对应不同激活方案，输出最优或次优策略组合。

输出：

- 每个沉睡资产的推荐激活路径。
- 每个策略的预期使用提升。
- 每个治理场景的数据包推荐。
- 平台整体优化建议。

## 五、预期结果与可视化产出

### 5.1 预期发现

项目预期能得到以下结果：

- 公共数据使用表现并不只由下载入口决定，数据主题、API、更新频率、字段说明和可组合性都会影响使用。
- 一部分数据集虽然当前下载和浏览低，但具备高场景价值和高组合价值，属于高潜低用资源。
- 沉睡资产不是同一种类型，可能分别由质量缺陷、接口缺失、语义模糊、更新滞后或场景孤岛导致。
- 数据组合推荐能把单个低价值数据转化为面向治理场景的数据包。
- 对不同类型沉睡资产，应采用差异化激活策略。

### 5.2 主要图表

建议输出：

- 公共数据主题分布图。

- 下载量/浏览量长尾分布图。
- 开放质量雷达图。
- 使用表现预测模型 SHAP 图。
- 沉睡度排名表。

- 治理场景数据包推荐图。

## 七、项目时间表

建议按 6 周推进。

### 第 1 周：数据可行性验证

- 完成上海公共数据开放平台结构调研。
- 确认下载量、浏览量、API、字段说明等可采集字段。
- 采集 100-300 个样本数据集做试验。
- 确定最终题名和数据字段表。

阶段目标：

- 证明数据能拿到。

## 八、风险与应对

### 风险 1：平台无法获取 API 调用量

应对：

- 使用下载量、浏览量、是否提供 API、是否关联应用作为替代指标。
- 将“使用表现”表述为可观测使用热度，而不是完整使用行为。

### 风险 2：无法证明数据被治理决策采用

应对：

- 不把治理采用作为主因变量。
- 改为研究“高潜低用识别”和“场景化激活”。
- 对典型应用关联只作为强证据样本，不作为全量判定标准。

### 风险 3：数据内容无法批量下载

应对：

- 优先诊断元数据质量。
- 对内容质量采用抽样下载。
- 将内容质量作为增强模块，而不是主模型唯一基础。

### 风险 4：推荐结果显得主观

应对：

- 结合文本相似度、字段连接规则、空间/时间字段可连接性和图模型共同推荐。
- 对推荐结果做典型案例人工核验。

### 风险 5：策略模拟因果性不足

应对：

- 明确称为反事实预测或预期增益估计，不声称严格因果。
- 使用相似数据集匹配和稳健性检验增强说服力。
